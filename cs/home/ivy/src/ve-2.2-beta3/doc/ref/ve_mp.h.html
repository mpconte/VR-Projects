The "ve_mp" module provides multi-processing support for the VE library. Multi-processing support is for distributing the rendering load amongst several tasks either for efficiency (e.g. multiple threads on a multi-pipe machine) or out of necessity (e.g. a distributed cluster hooked together by networks). It takes care of the following tasks:

  • creating and initializing slaves (local and remote processes, threads)
  • providing synchronization and data update methods

In any model, multi-processing works in a master-slave model. The initial process (i.e. the one from which the program is started) acts as the master, processes all incoming device events and pushes data (generally state updates) to the slaves.

The MP support is integrated in such a way that existing programs should work in their current single-process environments with nothing more than a recompile. To make a program MP-compatible, you will need to:

  • guard any program initialization with a call to veMPIsMaster(). For example:
    if (veMPIsMaster()) veDeviceAddCallbck(...)
    Such guard is not built-in to the various functions (e.g. veDeviceAddCallback) to avoid too much interdependence within the library's modules and to discourage code which would be incompatible with possible future models which would support more functionality in slaves.
  • arrange a data handler (see veMPDataAddHandler()) and push state and model information out to slaves. The library will push initial environment and profile information to the slaves itself. A simpler model in many programs is to simply set up state variables (see veMPAddStateVar()) which is a chunk of memory that will be automatically updated on all slaves at the beginning of each frame.

The MP layer has been reworked significantly for VE 2.2. In particular slaves are now created opportunistically using the veMPGetSlave() call to retrieve connection information for slaves. This makes the whole layer much more flexible and less dependent upon a particular purpose (e.g. the rendering subsystem).

Type: VeMPDataHandler
The type for an application MP callback. This allows functions in the application to be called for registered message types. (see veMPDataAddHandler).
typedef void (*VeMPDataHandler)(int tag, void *data, int dlen);

Structure: VeMPPkt
Describes a single communication packet (in either direction).
typedef struct ve_mp_impl_pkt {

seq - A monotonically increasing sequence number. These are generally assigned and interpreted by the platform-independent layer. The implementation layer should pass them unchanged.
unsigned long seq;

ch - Specifies the channel over which this data should be sent. Every implementation must provide at least a VE_MP_RELIABLE channel (see below). For the sending side of an implementation, if the specified channel cannot be used or does not exist, then the VE_MP_RELIABLE channel must be used. For the receiving side, the channel over which this packet was received should be specified here.
int ch;

msg - The message type id (as specified in the platform-independent MP interface).
int msg;

tag - The tag id (as specified in the platform-independent MP interface).
int tag;

dlen - The size of the data buffer in bytes. This is the size of meaningful data in the buffer not necessarily the size of the space allocated for the buffer. If this value is 0, then data may be NULL. If this value is > 0 then data must not be NULL (except in one special case - see veMPPktDestroy() below.
int dlen;

data - The buffer containing the payload of the packet. This may be NULL if the size of the payload (dlen field) is 0. The sending side of an implementation should not assume that this piece of memory is safe beyond the call to the veMPImplSend() function. The receiving side of an implementation should always allocate a new piece of memory here and allow the library/application to handle the freeing of the memory at a later date.
void *data;
} VeMPPkt;

Type: VeMPPktHandler
The "new"-style callback for internal data handlers. It replaces the VeMPIntDataHandler type which is no longer supported.

In general, src identifies what the source of the message is. For slave callbacks, src is always VE_MPTARG_MASTER. On master callbacks, src identifies the slave from which the packet was received.

typedef void (*VeMPPktHandler)(int src, VeMPPkt *p);

Function: veMPGetSlave
int veMPGetSlave(char *node, char *process, int allow_fail);
When called on the master, returns the identifier for the given slave. If no connection for the given slave exists then VE attempts to establish one. For any given slave connection (identified by a (node,process) pair) once a connection is established its identifier will not change for the duration of the program's execution. Thus, it is safe and valid for calling procedures to cache the return value of this function for a particular (node,process) pair.

Currently the VE communication system does not allow for communication between slaves. On slaves, this function will always indicate failure.

node - The name of the node on which the slave should reside. The value "auto" or the NULL pointer here can be used to indicate the node on which the master is running.
process - The name of the process in which the slave should reside. Process names are abstract. Slaves that share the same process name on the same node will run within the same process. The value "auto" is valid for all nodes. For the master node, "auto" refers to the master process. So an MP setup in which all slaves have nodes set to "auto" and processes set to "auto" is actually a single process (but possibly multi-threaded) program. The NULL pointer can be used in place of "auto".

The reserved name "unique" can be used here to indicate that a process that is not shared by any other slave should be created. Note that in this case the calling procedure must save the returned identifier - a subsequent call to this function with the process name "unique" will generate a new, different slave process.

allow_fail - If this is 0, then failure to establish a connection to a slave will be considered a fatal error. If this is non-zero then failure to establish a connection will not print an error message and return control to the calling procedure (with an error code).
A slave connection identifier which is >= 0 on success, or a value < 0 on error.

Function: veMPSlaveInit
int veMPSlaveInit(int *argc, char **argv);
Initializes the multi-processing for a slave. This needs to be called before any other initialization in the system - either as the first call in veInit() or before any other VE initialization. This is crucial as veMPSlaveInit() looks at the initial arguments to determine whether or not this is a slave process. This call removes any arguments that are specific to the MP module from argv and updates argc accordingly.

If this process is the master, then this function is used to cache the name of the program and the right set of arguments to use when creating slaves. That is, this function should be called exactly once for all processes, including master and slaves.

argc - A pointer to an integer containing the number of arguments in argv (as would be passed to main()).
argv - An array of char pointers representing strings (as would be passed to main()). Element 0 is assumed to be the program name and is left untouched when processing options.
Returns: 0 on success, non-zero on error. Note that the contents of argc and argv will be modified by this call.

Function: veMPInit
int veMPInit(void);
Initializes the MP subsystem on the master process creating the slave processes. This function should not be called before an environment has been properly defined. It is usually called automatically as part of the call to veRun(). On the master it starts the slaves and sends initialization information to them. On all processes, it starts up a thread for handling incoming messages.
Returns: 0 on success, non-zero on failure.

Function: veMPDataAddHandler
int veMPDataAddHandler(int tag, VeMPDataHandler handler);
Adds a callback to handle data updates.
tag - If this is VE_DTAG_ANY then any incoming application data message is passed to the callback. Otherwise, only those messages with the same tag as tag are passed. This call functions both on the master and on the slaves (i.e. the master can receive its own data push messages).
handler - The handler to be called when an appropriate message is received.
Returns: 0 on success, non-zero on failure.

Function: veMPDataPush
int veMPDataPush(int tag, void *data, int dlen);
Pushes data from the master process to the slave processes. This function can only be called on the master process.
tag - An identifier that will be passed to the callback on the slaves to help identify the contents of the message.
data - The message which is dlen bytes long.
dlen - The size of the message (in bytes) that is stored in data.
Returns: 0 on success, non-zero on failure.

Function: veMPLocationPush
int veMPLocationPush(void);
Pushes the origin and default eye information to the slaves. This must be done after these values are updated by the program.
Returns: 0 on success, non-zero on failure.

Function: veMPEnvPush
int veMPEnvPush(void);
Pushes the environment information to the slaves. This is typically done automatically by VE's initialization. The environment data is always pushed over the VE_MP_RELIABLE interface.
Returns: 0 on success, non-zero on failure.

Function: veMPProfilePush
int veMPProfilePush(void);
Pushes the user profile information to the slaves. This is typically done automatically by VE's initialization. The profile data is always pushed over the VE_MP_RELIABLE interface.
Returns: 0 on success, non-zero on failure.

Function: veMPAddStateVar
int veMPAddStateVar(int tag, void *var, int vlen, int flags);
Adds a state variable. A state variable is a piece of memory that will be passed from master to slave(s), typically indicating some state of the application. In order for state variables to be pushed correctly, this function must be called in the master and in all slaves. It is best for state variables to be added before the call to veRun(). Applications are responsible for synchronization between master and slaves if state variables are created after the call to veRun().
tag - An identifier that is used to identify this particular state variable. This value must be unique to this variable and be the same on both the master and slaves. The value must also be greater than or equal to 0.
var - A pointer to the state variable. This must point to a static storage location. If this is the address of a dynamic variable, grave problems with the application will likely occur.
vlen - The size of the state variable in bytes.
flags - Controls how the state variable is handled. Multiple flags can be combined with the bitwise-or (|) operator. Currently the following values are defined:
  • VE_MP_AUTO - state variable is automatically pushed before rendering each frame. If this flag is defined then the application does not need to explicitly push the state variable to the slaves; it will be done automatically. Without this flag, state variables need to be explicitly pushed to slaves with the veMPPushStateVar() function.
Returns: 0 on success, non-zero on failure.

Function: veMPPushStateVar
int veMPPushStateVar(int tag, int flags);
Distributes state variables to slaves. This function only has an effect on the master process. On slave processes, it is a no-op.
tag - Indicates which state variable to push. If this value is VE_DTAG_ANY, then all state variables are pushed, provided that they meet any constraints imposed by the flags (see below).
flags - Controls how variables are pushed. Multiple flags can be combined with the bitwise-or (|) operator. If this value is 0, then no constraints are made upon which variables are pushed. Currently the following values are defined:
  • VE_MP_AUTO - only state variables created with the VE_MP_AUTO flag will be pushed. Applications should not need to use this flag - it is for internal use.
Returns: 0 on success, non-zero on failure.

Function: veMPIsMaster
int veMPIsMaster(void);
A predicate function that tests whether or not the current process is the master.
Returns: 1 if this is the master process, 0 otherwise.

Function: veMPGetSlaveGuard
int veMPGetSlaveGuard(void);
Returns the current state of the slave guard (on or off). When the slave guard is on (the default), then a number of VE functions are limited to "master-only" use - that is, when they are called on a slave, they act as no-ops. Applications can control this behaviour with the veMPSetSlaveGuard() function. The goal of this mechanism is to limit the need for the use of veMPIsMaster() in application code. Functions that are guarded include:
  • veAddTimerProc()
  • veDeviceAddCallback()
  • veDeviceApplyEvent()
  • veDeviceInsertEvent()
  • veDeviceProcessEvent()
  • veDevicePushEvent()
  • veDeviceUse()
  • veDeviceUseByDesc()
  • veMPDataPush()
  • veMPGetResponse()
  • veMPLocationPush()
  • veMPPushStateVar()
Specific implementations may also guard other functions. For example, in the OpenGL implementations, the following functions are also guarded:
  • veiGlSetPreDisplayCback
  • veiGlSetPostDisplayCback

Function: veMPSetSlaveGuard
void veMPSetSlaveGuard(int state);
Sets the state of the slave guard (see veMPGetSlaveGuard()). A non-zero value turns the slave guard on, a zero value turns it off.
state - The new state of the slave guard.

Function: veMPTestSlaveGuard
int veMPTestSlaveGuard(void);
Returns a non-zero value if the guard is active and this is a slave process.

Function: veMPId
int veMPId(void);
Returns a unique id (within this VE application) of the current process. This can be used to uniquely identify a process within VE. Applications can assume that this value (say i) is such that 0 <= i < veMPNumProcesses(), but should not assume a specific meaning of any one value. Specifically, one should not assume that id "0" is the master. Use the veMPIsMaster() call instead.
Returns: The MP id of this process, or -1 if this is an asynchronous slave.

Function: veMPNumProcesses
int veMPNumProcesses(void);
Returns: The number of processes in this VE application. Note that this will be a tight upper-bound, but not necessarily exact. In situations where slaves can tolerably vanish, this value may be inexact.

Function: veMPSlaveProcess
char *veMPSlaveProcess(int k);
k - The slave identifier, must be >= 0 and < veMPNumProcesses(). This function is only valid on the master node. This function exists primarily for use during initialization of MP sub-modules.

On slave nodes, the function is meaningful if called with the argument VE_MP_SELF in which case, the process name of the calling slave will be returned.

Returns: A read-only string containing the name of this slave's process. For invalid slave identifiers, the result will be NULL.

Function: veMPSlaveNode
char *veMPSlaveNode(int k);
k - The slave identifier, must be >= 0 and < veMPNumProcesses(). This function is only valid on the master node. This function exists primarily for use during initialization of MP sub-modules.

On slave nodes, the function is meaningful if called with the argument VE_MP_SELF in which case, the node name of the calling slave will be returned.

Returns: A read-only string containing the name of this slave's node. For invalid slave identifiers, the result will be NULL.

Function: veMPCleanup
void veMPCleanup(void);
Attempts to cleanly shutdown the multi-processing subsystem and terminate any slave processes. The MP module will generally set this up to be called on exit.

Function: veMPSendMsg
int veMPSendMsg(int ch, int target, int msg, int tag, void *data, int dlen);
A generic message send function. This is only meant for internal use in VE and should not be used by applications. It supercedes veMPSendCtrl() (which is no longer available).
ch - The channel over which to send. Must be VE_MP_RELIABLE or VE_MP_FAST.
target - The slave id to which to send the message. There are two reserved values:
  • VE_MPTARG_ALL - is a broadcast from the master to all slaves
  • VE_MPTARG_MASTER - for the special case of a slave sending a message to the master
msg - The message-type identifier.
tag - The message tag.
data - The body of data. This value can be NULL iff dlen is 0.
dlen - The size of the body of data in bytes.
Returns: 0 if successful, non-zero on failure.

Function: veMPRenderFrame
void veMPRenderFrame(long frame);
Gets the MP layer to render a frame by communicating with slaves.
frame - Specifies the frame number to generate. If this value is negative, then a frame number will be chosen randomly. Unless you really know what you are doing, you should generally not specify a frame number - i.e. always pass '-1' as this value.

MP Rendering

This section is an extension to existing control messages. The MP rendering has been redesigned from the simple VE 2.1 interface. This change requires that all rendering implementations (library implementations, not applications) be updated to deal with the new control messages. This spec extends the RENDER and SWAP messages.

The master and all slaves maintain their own frame counter which should be initialized to 0. In addition, each slave will keep a swapped flag which is initialized to 1. The first frame that will be actually rendered will be frame 1. The master may re-initialize the frame counter to avoid overflow at anytime. However, it is guaranteed that two subsequent frames will never share the same frame number. (This guarantee may be extended in the future to gurantee uniqueness over n frames if necessary.) The idea is that the master sends a steady stream of RENDER and SWAP messages with ever increasing (and occasionally resetting) frame numbers. Slaves respond to these packets with information about their own state. If a packet represents a "future" state for the slave, then the slave advances to that state (if possible) and updates its own state accordingly. This way, if a message is lost or delivered out-of-order, the system should hold together.

MP Implementation

The following section describes the implementation-abstraction interface for multi-processing under VE. This interface is not used by applications and developers using VE typically do not need to consider the following information. However, developers extending or porting VE should be aware of the following interface.

The core ve_mp module is platform-independent. The low-level details of how to instantiate slaves and the transport of data between nodes is left up to the implementation layer. All implementation layers should provide the following interface.

Packets

This section describes the abstract structure of packets that are sent on the wire. Implementations are free to encode the packets in anyway they see fit and may add information as needed, but must present the data in the following structure to the platform-independent level.

Function: veMPPktCreate
VeMPPkt *veMPPktCreate(int dlen);
Creates a new packet with appropriately allocated data space.
dlen - The size of the data buffer to allocate. If this is <= 0 then no data buffer is allocated (data member will be NULL).
Returns: A newly allocated packet structure. Except for dlen and data, all other fields will be initialized to '0' (or its equivalent).

Function: veMPPktDestroy
void veMPPktDestroy(VeMPPkt *p);
Frees a previously allocated packet. This is the one case where it is valid for a packet to have a NULL value for the data member, but a non-zero value for the dlen member. This allows a calling function to preserve the buffer space of the packet without copying its contents by saving the pointer elsewhere and setting the data member of the packet to NULL (to prevent this function from freeing the buffer).
pkt - The packet to destroy. All memory associated with the packet (including its buffer space) is freed. See the description above for details on preserving the buffer space.

Type: VeMPImplConn
An opaque type representing a connection. The platform-independent layer assumes that this is a pointer-like value, in that it can be compared against NULL and that NULL is never a valid connection.
typedef void *VeMPImplConn;

Function: veMPImplRecv
int veMPImplRecv(VeMPImplConn c, VeMPPkt **p_r, long tmout);
Recieves a packet from a connection.
c - The connection over which to receive a packet - must not be NULL.
p_r - A pointer to a VeMPPkt pointer. If a packet is received successfully then a new packet structure is allocated and the pointer to this structure is stored here. Otherwise, this value is set to NULL.
tmout - A time-out. If this value is > 0 then the function call will wait at most this many microseconds before returning. Note that it may wait longer due to operating-system implementation constraints. If this value is equal to 0 then the function call will return immediately if no packet is available. If this value is < 0 then the function will wait indefinitely for a packet (i.e. time-out is disabled).
Returns: 0 on successful reception of a packet, < 0 if an error occurred on the connection while reading, or > 0 if the function timed-out while waiting for a packet to arrive.

Function: veMPImplSysdep
int veMPImplSysdep(VeMPImplConn c, VeMPPkt *p);
System-dependent callback. Allows implementations to use the VE_MPMSG__SYSDEP message type to pass information between master and slave when needed. Usually called by the platform-independent MP layer whenever a "SYSDEP" message is received.

Function: veMPImplWait
int veMPImplWait(VeMPImplConn *c, VeMPImplConn *c_res, int n);
Waits for data to become available on one or more of the connections in the set.
c - An array of size n. Some entries may be NULL (these will be ignored). This is the input set of connections to wait upon.
c_res - An array of size n. This array will be updated with the set of connections upon which data is available. After calling this function, some or all entries may be NULL. Non-NULL entries are those connections with real data. Their positions will be the same as the positions in the input array.
n - The size of the c and c_res arrays.
Returns: 0 on success, non-zero on failure.
